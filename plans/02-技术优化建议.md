# 技术优化建议

## 后端性能优化

### 1. LLM 结果缓存分层 ⭐⭐⭐⭐

**现状**：每次 AI 解读都实时调用 LLM，响应时间 3-5 秒，成本高。

**优化方案**：
- 引入 `moka-rs`（纯 Rust 内存缓存）缓存热点 LLM 输出
- 缓存 Key：报告 ID + 解读类型 + 模型版本
- TTL：AI 解读 7 天，名称标准化 30 天
- 支持手动清除缓存（报告更新时自动失效）
- 启动时预热常用患者的解读缓存

**预期收益**：热查询延迟从 3-5s → <100ms，API 调用成本降低 60%

---

### 2. LLM 多模型降级策略 ⭐⭐⭐⭐

**现状**：单一 API 提供商，服务故障时功能完全不可用。

**优化方案**：
- 支持多提供商配置（Qwen / Claude / GLM / 本地 Ollama）
- 熔断器模式：连续失败 3 次后自动切换备用提供商
- 降级策略：
  - OCR 失败 → 提示用户手动录入
  - 解读失败 → 展示上次缓存结果 + 提示刷新
  - 标准化失败 → 保留原始名称，人工确认
- 本地 Ollama 支持：零数据出境的私密模式

**预期收益**：可用性从 95% → 99.5%

---

### 3. 数据库查询优化 ⭐⭐⭐

**现状**：部分 API 存在 N+1 查询（如 `list_reports` 逐条加载 `test_items`），批量操作未合并事务。

**优化方案**：

```sql
-- 推荐添加的索引
CREATE INDEX idx_reports_patient_date ON reports(patient_id, report_date DESC);
CREATE INDEX idx_test_items_status ON test_items(status, report_id);
CREATE INDEX idx_temperatures_patient_date ON temperatures(patient_id, measured_at DESC);
CREATE INDEX idx_medications_patient ON medications(patient_id, start_date DESC);
```

- 报告列表 API 改为 JOIN 一次性加载 `test_items`
- 批量操作（批量更新状态、批量删除）合并为单个事务
- 为 `patients` 表的搜索字段添加全文索引

**预期收益**：列表查询时间降低 30-50%

---

### 4. 搜索引擎升级 ⭐⭐⭐

**现状**：基于 SQLite LIKE 的简单搜索，不支持模糊匹配和拼音。

**优化方案**：
- 引入 **Meilisearch**（轻量、Rust 实现、开箱即用）
- 支持：拼音首字母 + 模糊匹配 + 中文分词 + 错别字容错
- 增量同步：患者/报告更新时异步推送到 Meilisearch
- 电话号码支持模糊匹配（少记几位也能找到）

**替代方案**：Tantivy（嵌入式，零依赖，适合单机部署）

**预期收益**：搜索准确率 +30%，响应延迟 <50ms

---

### 5. 大文件异步流式处理 ⭐⭐⭐

**现状**：大图片/PDF 上传时同步处理，可能导致内存压力和请求超时。

**优化方案**：
- 流式 multipart 处理：边接收边写磁盘，不缓存整个文件到内存
- 大文件 OCR 分页调用：超过 5MB 的图片分块处理
- 上传队列：前端支持后台上传，显示进度条，不阻塞用户操作
- 异步任务状态轮询：`GET /api/tasks/{task_id}` 查询处理进度

**预期收益**：支持 500MB 文件，内存占用降低 60%

---

### 6. 健康监控与可观测性 ⭐⭐⭐

**现状**：`/api/health` 端点基础，缺乏详细指标。

**优化方案**：

增强健康检查端点：
```json
{
  "status": "ok",
  "db": { "latency_ms": 2, "connections": 5 },
  "llm_providers": { "qwen": "ok", "siliconflow": "degraded" },
  "cache": { "hit_rate": "87%", "memory_mb": 128 },
  "uptime_seconds": 86400
}
```

- 集成 Prometheus 指标（请求延迟、错误率、LLM 调用次数）
- 结构化日志（JSON 格式）接入 ELK / Loki
- 关键操作审计日志异步写入（不影响主链路）

**预期收益**：故障发现时间从小时级 → 分钟级

---

### 7. 数据一致性强化 ⭐⭐⭐

**优化方案**：
- 关键写操作（创建报告 + 保存检验项 + 触发标准化）使用显式事务
- 乐观锁：`updated_at` 字段防止并发冲突（"先读后写"场景）
- 死锁检测：设置 SQLite `busy_timeout`，失败时自动重试（最多 3 次）
- 定期数据一致性检查任务（孤儿 test_items、缺失 patient 引用等）

---

### 8. API 版本管理 ⭐⭐

**优化方案**：
- 通过 `Accept: application/json; version=2` 头进行版本协商
- 明确弃用策略：废弃字段提前 3 个月在响应头中提示（`Deprecation: true`）
- 为未来多客户端（Web / iOS / 第三方）预留版本兼容空间

---

### 9. 部署与扩展性 ⭐⭐

**优化方案**：
- 完善 Dockerfile：多阶段构建，最终镜像 <50MB
- 提供 `docker-compose.yml`：一键启动（含 Meilisearch、Redis 可选）
- ORM 抽象层：为未来 SQLite → PostgreSQL 迁移预留接口
- K8s Helm Chart：支持水平扩展（OCR / LLM 作为独立微服务）

**预期收益**：支持并发从 100 → 10000 用户

---

## 前端性能优化

### 1. 并行数据请求 ⭐⭐⭐⭐

**现状**：PatientDetail 页面串行加载 patient → reports → temperatures → expenses。

**优化方案**：
```typescript
// 改为并行加载
const [patient, reports, temperatures] = await Promise.all([
  api.getPatient(id),
  api.getReports(id),
  api.getTemperatures(id),
]);
```

在 `api/client.ts` 增加简单缓存层（5 分钟 TTL），切换患者时命中缓存。

**预期收益**：首屏加载时间降低 40-60%

---

### 2. 路由预加载 ⭐⭐⭐

**优化方案**：
- 在 Dashboard 鼠标悬停患者卡片时，预加载 PatientDetail 的代码 chunk
- 使用 `<link rel="prefetch">` 预加载高频访问页面
- 相邻路由代码在空闲时间后台预加载

**预期收益**：路由切换延迟降低 30%

---

### 3. 图表渲染性能 ⭐⭐⭐

**现状**：手写 SVG 图表在数据点 >500 时可能卡顿。

**优化方案**：
- 数据点 >200 时自动启用数据聚合（相邻点合并为均值）
- 考虑引入 **ECharts**（性能好、医疗图表模板丰富）替代手写 SVG
- 超长表格启用虚拟滚动（Virtual Scrolling）

---

### 4. Bundle 体积优化 ⭐⭐

**优化方案**：
- 用 `vite-bundle-visualizer` 分析 chunk 大小，找出体积异常的依赖
- SVG 图标合并为 symbol sprite，减少 HTTP 请求
- 确认 TailwindCSS v4 的 purge 配置正确，删除未使用样式
- 图片资源使用 WebP 格式

**预期收益**：首屏 bundle 大小降低 20-30%

---

### 5. PWA 缓存策略明确化 ⭐⭐

**优化方案**（使用 Workbox）：

| 资源类型 | 策略 |
|----------|------|
| HTML / JS / CSS | Cache First（版本化文件名） |
| API 响应（列表类） | Stale While Revalidate |
| API 响应（写操作） | Network Only |
| 图片（上传的报告图） | Cache First（长缓存） |

- 新版本发布时，`ReloadPrompt` 弹窗提示用户更新
- 使用 IndexedDB 离线缓存患者列表，断网时可浏览

---

## 用户体验优化

### 1. 导航结构重构

**问题**：患者详情子页面（timeline / medications / health-assessment 等）靠 URL 跳转，缺乏清晰的导航层次。

**优化方案**：
- 患者详情页顶部增加 Tab 导航栏（概览 / 报告 / 趋势 / 用药 / 体温 / 费用 / 时间线）
- 添加面包屑：`患者列表 > 张三 > 报告详情`
- 记住用户上次访问的 Tab，下次进入时自动还原

---

### 2. 移动端体验适配

**问题**：趋势图、报告对比在手机竖屏下显示不佳。

**优化方案**：
- TrendChart / TemperatureChart 支持横向滑动查看
- ReportCompare 在移动端改为堆叠式布局（上下排列而非左右）
- 患者列表卡片在手机上精简，仅显示姓名、最近就诊、风险等级

---

### 3. 视觉反馈强化

**优化方案**：
- 检验值异常时：左侧彩色竖条（正常=绿 / 偏高偏低=黄 / 危急=红）+ 背景色
- 未处理事项：导航图标右上角红色徽章数字
- 操作成功/失败：Toast 通知位置统一到右上角，自动消失

---

### 4. 表单体验提升

**优化方案**：
- 字段实时验证（失去焦点时立即显示绿勾 / 红叉）
- 提交时错误字段高亮 + 页面顶部生成错误汇总列表
- 身份证格式错误时提示正确格式示例
- 长表单支持草稿自动保存（localStorage）

---

### 5. 数据操作增强

**优化方案**：
- 检验结果表格支持：列排序、列显示/隐藏、多行选择、批量修改状态
- 冻结首列（指标名称），横向滚动时始终可见
- 表格数据支持直接导出为 Excel（在现有 CSV 基础上）
